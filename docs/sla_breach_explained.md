# Что такое SLA Breach и зачем нужен алерт

## Определение

**SLA** (Service Level Agreement) — это договорённость о времени реакции. В контексте лидов это конкретный дедлайн:

> *«Менеджер должен связаться с лидом в течение X часов после его поступления»*

**SLA Breach** = этот дедлайн нарушен. Лид пришёл, обработался системой, черновик ответа готов — но менеджер так и не нажал **«Approve & Send»**.

---

## Почему это критично в B2B продажах

Исследования показывают: если ответить на входящий запрос в первые **5 минут** — конверсия в звонок в **21 раз выше**, чем если ответить через 30 минут. Через 24 часа лид, скорее всего, уже разговаривает с конкурентом.

| Время реакции | Вероятность выхода на контакт |
|---------------|-------------------------------|
| < 5 минут | Максимальная (×21 vs 30 мин) |
| 30 минут | Резкое падение конверсии |
| > 24 часов | Лид с высокой вероятностью потерян |

---

## Как это работает в системе

```
Лид поступил → next_action_due_at = сейчас + SLA_HOURS (по умолчанию 4 ч)
                                              │
                               ┌──────────────┘
                               ▼
                    Каждые 15 минут: check_sla()
                               │
               ┌───────────────┴──────────────────┐
               │ next_action_due_at в прошлом?     │ нет → всё ок
               │ да
               ▼
    Telegram: 🔴 SLA Breach → "Petra Novak, Industrials EU — просрочено!"
    event_log: alert_sla_breached → { owner, status, next_action_due_at }
```

Пример Telegram-уведомления:

```
🔴 SLA Breach
Lead: Petra Novak (Industrials EU)
Owner: oleg@workshop.ai
Status: new
Due: 2000-01-01T00:00:00+00:00
ID: <lead_id>
```

## Зачем алерт, а не просто пометка в базе

| Без алерта | С алертом |
|------------|-----------|
| Менеджер узнает только если сам зайдёт на дашборд | Получает пуш в Telegram немедленно |
| Лиды теряются тихо, без видимых последствий | Каждое нарушение видно и зафиксировано |
| Нет персональной ответственности | В алерте указан конкретный `owner` |
| Руководитель не знает о системных проблемах | Можно добавить руководителя в `TELEGRAM_CHAT_IDS` |

---

## Защита от спама алертами

Если лид завис на несколько дней, система **не будет слать одно и то же сообщение каждые 15 минут**.

Повторный алерт по одному и тому же лиду блокируется на **6 часов** — система проверяет наличие события `alert_sla_breached` в истории и пропускает итерацию, если последний алерт был менее 6 часов назад.

```python
# app/services/sla.py — логика подавления дублей
hours_since = (datetime.now(timezone.utc) - last_alert_dt).total_seconds() / 3600
if hours_since < 6:
    continue  # Already alerted recently
```

## Конфигурация

В файле `.env`:

```env
# Дедлайн реакции на лид (в часах)
SLA_HOURS=4

# Интервал проверки SLA (в секундах, по умолчанию 900 = 15 минут)
SLA_CHECK_INTERVAL_SECONDS=900
```

Добавить дополнительных получателей алерта:

```env
# Несколько chat_id через запятую
TELEGRAM_CHAT_IDS=236883036,987654321
```

---

## Ручной триггер (для демо и тестирования)

```
POST /api/v1/leads/{lead_id}/sla-check
```

Этот эндпоинт немедленно сдвигает `next_action_due_at` лида в прошлое и запускает проверку — удобно для демонстрации без ожидания реального дедлайна.

**Ответ:**
```json
{
  "ok": true,
  "breached_count": 1
}
```
